# 缺陷管理系统项目说明

## 一、项目背景

本项目是学校期末作业项目，目标是完成一个可运行、可部署、可截图展示的缺陷管理网页系统。系统部署目标为鲲鹏 ARM 架构服务器，服务器系统可使用 openEuler，并通过 Docker 容器打包运行。

系统围绕软件测试和开发协作中的 Bug 流转过程设计，覆盖缺陷提交、处理、修复、关闭和操作留痕等核心流程。

## 二、需求分析

系统需要满足以下要求：

- 用户可以通过账号密码登录。
- 系统有管理员、测试人员、开发人员三种角色。
- 不同角色看到的菜单和可执行操作不同。
- 测试人员可以新增 Bug、查看自己提交的 Bug、关闭已修复 Bug。
- 开发人员可以查看分配给自己的 Bug、修改状态为已修复、转交 Bug。
- 管理员可以查看和管理全部 Bug、用户和统计数据。
- 每次关键操作都要写入操作记录。
- 系统需要提供简单统计图表，用于项目书截图。
- 项目最终支持 Docker Compose 一键启动。

## 四、系统架构

系统采用前后端分离架构：

```text
浏览器
  │
  ▼
frontend 容器：Nginx + Vue 静态资源，端口 8080
  │ /api 代理
  ▼
backend 容器：Spring Boot API，端口 8081
  │ JDBC
  ▼
mysql 容器：MySQL 8，端口 3306
```

前端负责页面展示、表单交互、角色菜单和接口调用。后端负责登录认证、权限判断、业务处理、数据库读写和统一接口返回。MySQL 保存用户、缺陷和操作记录。

## 六、功能模块

### 1. 登录模块

用户输入用户名和密码，登录成功后后端返回 token 和用户信息。前端将 token 保存到浏览器本地存储，请求接口时通过 `Authorization: Bearer token` 携带。

### 2. 控制台首页

展示 Bug 总数、今日新增、未处理、已修复、已关闭、严重 Bug 数量，并使用 ECharts 展示状态统计和严重程度统计。

### 3. Bug 管理

支持新增、编辑、删除、列表查看、详情查看、搜索和筛选。Bug 字段包括标题、所属模块、严重程度、问题描述、复现步骤、截图地址、状态、创建人、处理人。

### 4. 状态流转

系统核心流程为：

```text
已提交 → 已修复 → 已关闭
```

状态修改会自动写入操作记录。

### 5. 转交处理人

管理员和开发人员可以把 Bug 转交给其他用户处理。转交时保存旧处理人、新处理人和备注。

### 6. 操作记录

系统记录新增 Bug、编辑 Bug、修改状态、转交处理人、删除 Bug、关闭 Bug 等操作，并在详情页以时间线方式展示。

### 7. 统计分析

统计页面展示总数、今日新增、未处理、已修复、已关闭、严重 Bug 数量，并展示状态和严重程度图表。

## 七、数据库设计

数据库名：`bug_management`

## 八、接口说明

统一前缀：`/api`

统一返回格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/auth/login` | 登录 |
| GET | `/api/auth/me` | 当前用户 |
| GET | `/api/users` | 用户列表 |
| GET | `/api/bugs` | Bug 列表 |
| GET | `/api/bugs/{id}` | Bug 详情 |
| POST | `/api/bugs` | 新增 Bug |
| PUT | `/api/bugs/{id}` | 编辑 Bug |
| DELETE | `/api/bugs/{id}` | 删除 Bug |
| PUT | `/api/bugs/{id}/status` | 修改状态 |
| PUT | `/api/bugs/{id}/assign` | 转交处理人 |
| GET | `/api/bugs/{id}/logs` | 操作记录 |
| GET | `/api/statistics/overview` | 统计数据 |
